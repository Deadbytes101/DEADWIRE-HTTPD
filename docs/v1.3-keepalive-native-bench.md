# V1.3 KEEP-ALIVE NATIVE BENCH

Branch: `feat/v1.3-keepalive`

Server binary:

```txt
build/deadwire_keepalive_experimental.exe
```

Client:

```txt
tools/deadwire_bench.c --keepalive
```

Mode shape:

```txt
mode=close      one request per TCP connection
mode=keepalive  one TCP connection per round, many sequential requests on that socket
```

This is the first apples-to-apples client comparison path because the close and keep-alive modes use the same native C benchmark client.

## Command shape

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/build-win-keepalive-experimental.ps1
$exe = (Resolve-Path .\build\deadwire_keepalive_experimental.exe).Path
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/bench-native.ps1 -ServerExePath $exe -Port 19850 -Path /health -Requests 32768 -Rounds 5 -KeepAlive
```

## Comparison against V1.2 quiet native baseline

Baseline used here is the final clean V1.2 quiet run:

```txt
/health              median_rps=6570.15 median_avg_ms=0.152
/missing-bench.txt   median_rps=5700.92 median_avg_ms=0.175
/hello.txt           median_rps=5106.36 median_avg_ms=0.196
/                    median_rps=5147.70 median_avg_ms=0.194
```

Estimated keep-alive gain:

```txt
/health              rps +28.4% avg_ms -21.7%
/hello.txt           rps +11.4% avg_ms -10.2%
/missing-bench.txt   rps +15.5% avg_ms -13.1%
/                    rps +20.3% avg_ms -17.0%
```

## Interpretation

```txt
native keep-alive wins against the V1.2 quiet native close-after-response baseline on all tested paths
/health shows the largest gain because it avoids repeated connection lifecycle cost on the smallest response
static and index still win despite keeping the simple blocking single-threaded server model
this is not a concurrency win; it is a connection-reuse win
```

## Guardrail

The default binary should remain close-after-response until this experimental path is promoted deliberately.
