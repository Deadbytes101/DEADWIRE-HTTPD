# V1.3 KEEP-ALIVE PROMOTION PLAN

Branch: `feat/v1.3-keepalive`

## Current state

The V1.3 keep-alive path is implemented as an experimental generated Windows build:

```txt
build/deadwire_keepalive_experimental.exe
```

The default binary remains unchanged:

```txt
build/deadwire.exe
```

Current behavior split:

```txt
default binary:      close-after-response
experimental binary: keep-alive loop + Connection: keep-alive header
```

## Promotion options

### Option A: keep as experimental build flavor

```txt
make build-keepalive-experimental
make verify-keepalive-experimental
make bench-native-keepalive-experimental
```

Pros:

```txt
lowest risk
default behavior remains frozen
lets the project keep collecting data
```

Cons:

```txt
V1.3 feature is not the normal server path
users must opt in manually
```

### Option B: promote to a named stable build flavor

Rename the concept from experimental to stable flavor without changing the default binary:

```txt
make build-keepalive
make verify-keepalive
make bench-native-keepalive
```

Pros:

```txt
feature becomes first-class
low risk because default still remains unchanged
cleaner UX than experimental naming
```

Cons:

```txt
requires target/documentation cleanup
still keeps two Windows runtime variants
```

### Option C: promote keep-alive to default Windows behavior

Change the default generated Windows server to keep accepted sockets open for repeat requests and advertise:

```txt
Connection: keep-alive
```

Pros:

```txt
best default performance for sequential repeated requests
simpler user-facing binary story
```

Cons:

```txt
highest risk
changes HTTP connection semantics
requires stronger close-condition policy and more regression tests
```

## Recommended path

Use Option B first.

Promote the current experimental path into a named stable build flavor:

```txt
build/deadwire_keepalive.exe
```

with Make targets:

```powershell
make build-keepalive
make verify-keepalive
make bench-native-keepalive
```

Keep the default binary close-after-response until the keep-alive flavor has more stress coverage.

## Required before default promotion

Do not make keep-alive the default until these are done:

```txt
1. verify keep-alive close behavior when the client disconnects
2. verify malformed request behavior on an already-open connection
3. verify repeated 404/static/index paths for more than two requests
4. verify no accidental default behavior change
5. run same-session quiet vs keep-alive native benchmark again
```

Minimum promotion gate:

```powershell
make verify
make verify-quiet
make probe-keepalive
make verify-keepalive
make bench-native-quiet
make bench-native-keepalive
```

## Final classification

```txt
V1.3 keep-alive is promotion-ready as a stable build flavor.
It is not yet ready to become the default binary behavior.
This is a connection-reuse win, not a concurrency win.
```
