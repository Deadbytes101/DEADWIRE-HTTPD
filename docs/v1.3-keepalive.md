# V1.3 KEEP-ALIVE NOTES

Branch: `feat/v1.3-keepalive`

## Starting truth

Before changing the server loop, verify the current Windows server behavior with a single TCP connection and two HTTP requests.

Command:

```powershell
make probe-keepalive
```

Observed result:

```txt
probe-keepalive: close-after-response confirmed path=/health first_bytes=112
```

Interpretation:

```txt
current server behavior: close-after-response
request path tested:      /health
first response bytes:     112
keep-alive supported:     no
```

This is expected for the V1.2 server. V1.3 work should now target an experimental server loop that can handle more than one request on the same accepted socket.

## Experimental health-only loop

The first experimental build pass does not change the default binary. It builds a separate Windows executable:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/build-win-keepalive-experimental.ps1
```

Output:

```txt
build/deadwire_keepalive_experimental.exe
```

Probe it with:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/probe-keepalive-experimental.ps1
```

This first pass only loops the `/health` success path back to the request receive path. It is intentionally narrow. The default server remains close-after-response.

## Guardrail

Do not change the default server behavior blindly. Keep the baseline verified while building the new path:

```powershell
make verify
make verify-quiet
make probe-keepalive
```

A successful V1.3 keep-alive implementation should eventually flip the probe from:

```txt
close-after-response confirmed
```

to:

```txt
supported
```

Then benchmark it against the quiet one-request-per-connection baseline.
